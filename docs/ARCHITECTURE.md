# Project Architecture

*Currently in the scaffolding stage.*

## Core Principles
1. **Separation of Concerns**: Each component will handle one discrete part of the system.
2. **Environment Variables**: Local keys and strings are passed via the `.env` at runtime.
3. **Open Tools**: Built using fully accessible APIs and logic to ensure everything is debuggable and open-source.

## Technologies Used
- **Backend (Core Logic):** Rust via Tauri (Tokio, reqwest, rusqlite, etc.)
- **Frontend (UI):** React 19 + TypeScript + Vite.
- **State Management:** React Context (`WindowContext`) for OS-level window lifecycle; Zustand/LocalState for UI-specific reactivity.
- **Styling:** Vanilla CSS + CSS Modules + M3 Expressive Design tokens.

## Application Lifecycle & Bootstrap
- **Sovereign Mode Loader:** `App.tsx` acts as a high-level router (`ModeLoader`). It queries the Rust backend (`get_launch_mode`) to decide whether to mount:
  1. **`SovereignInstallerFlow`**: The custom React-based setup experience for new users.
  2. **`SovereignUninstallFlow`**: The deep-clean removal utility.
  3. **`MainDesktop`**: The core production environment.
- **Sovereign Bootstrap:** We eliminated standard NSIS/MSI installers in favor of a "Portable-First" React bootstrapper that handles executable placement, shortcut creation, and registry registration entirely in-app.

## Frontend Architecture & Window Management
- **`WindowContext`**: Centralized OS-style window management. All apps (Chat, Browser, Files, etc.) are managed as `WindowInstance` objects. The context exposes `openWindow`, `closeWindow`, and `toggleMinimize` methods across the entire component tree.
- **`MainDesktop`**: Extracted desktop logic that handles the rendering of the `DesktopEnvironment`, `XRAppBar`, `AppWindow` maps, and the `XRNavigationBar`.
- **`AppWindow`**: A high-level wrapper component providing draggable/resizable frames, window controls (Minimize/Close), and standard M3 elevation.

- **Model Storage:** Local models (`.gguf`) are searched for and stored in the application data directory (`~/.Magnolia/models`). The Rust backend (`tauri-plugin-fs`) provides utility functions to scan and index available models.
- **Engine Execution:** The Rust backend wraps `llama.cpp` directly (via sub-process or bindings), managing the lifecycle of the model inference. 
- **Streaming Tokens:** As the `llama.cpp` engine generates tokens, Rust captures the `stdout`/events over an async stream using `tokio`. These chunks are then emitted as Tauri window events (`emit`) from the Rust layer to the frontend.
- **Frontend State:** The React UI listens for these specific inference events and appends incoming tokens to the chat interface state in real-time, matching standard chat-bubble typography constraints.

## Data Privacy & API Key Management
- **Security First:** The application prioritizes local-first inference. No user data, chats, or settings are sent to external telemetry servers.
- **Key Storage:** API Keys for cloud failovers (OpenAI, Anthropic) or Hugging Face Hub downloads are managed exclusively by the Rust backend using OS-level secure storage (`keyring` crate) or local encrypted `.env` files.
- **Leak Prevention:** `.env` files and `.env.*` patterns (except `.env.example`) are strictly ignored via `.gitignore`. The UI relies solely on Tauri `invoke` commands to verify whether a key is present, ensuring keys never persist in the frontend bundle or React state.

## Chat History & State Management
- **Hybrid State Approach:** The application uses a backend-heavy architecture to manage Chat History.
- **Backend Persistence (Rust/SQLite):** Central to our "local-first" philosophy, all chat histories, user prompts, and metadata are persisted securely via a local SQLite database driven by the Rust backend. This enables high scalability for large chat logs and facilitates future enhancements like Full Text Search (FTS), Retrieval-Augmented Generation (RAG) over `.md` files, and local Knowledge Graphs.
- **Frontend State (Zustand):** The React frontend leverages Zustand strictly for *ephemeral, in-flight state*, such as buffering the streaming token output currently being generated by an LLM model, or tracking short-lived UI application state. It commits data back to the Rust backend to save interactions only upon sequence completion.

## Local RAG & Vector Embeddings
- **Embedding Generation:** Upon parsing local `.md` files via the indexer, the Rust backend invokes a lightweight, quantized embedding model (e.g., `all-MiniLM-L6-v2.gguf`) using our existing `llama.cpp` bindings. This generates high-quality float32 vector representations of the workspace knowledge.
- **Vector Storage (`sqlite-vec`):** Instead of introducing a heavy, separate Vector Database dependency like Pinecone or Chroma, embeddings are stored entirely within our existing SQLite database using the `sqlite-vec` extension. This creates a seamlessly integrated virtual table linking vector dimensions directly to our `Documents` and `Chats` tables.
- **Semantic Search:** When a user prompts the LLM, the raw prompt is instantly embedded, and Rust executes a rapid nearest-neighbor (KNN) search utilizing `sqlite-vec` before injecting the most relevant chunks into the LLM context window.
